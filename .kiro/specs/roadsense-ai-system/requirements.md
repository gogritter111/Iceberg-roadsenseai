# Requirements Document

## Introduction

RoadSense AI is an intelligent road infrastructure monitoring system that automatically detects, validates, and reports road damage incidents across Indian cities using AWS serverless architecture and Amazon Bedrock AI. The system collects signals from multiple data sources (RSS feeds, weather APIs), processes them through a multi-stage AI pipeline, and generates high-confidence incident reports for municipal authorities.

The system operates as a fully automated, event-driven pipeline that processes 5-10 signals per hour and generates 200-400 validated incidents per month with 92% classification accuracy and less than 8% false positive rate.

## Glossary

- **Signal**: A single data point from an external source (RSS feed, weather API) that may indicate a road infrastructure issue
- **Scraper_Lambda**: AWS Lambda function that collects signals from RSS feeds and weather APIs on an hourly schedule
- **Ingest_Lambda**: AWS Lambda function that generates embeddings and stores signals in vector database
- **Inference_Lambda**: AWS Lambda function that orchestrates the 5-stage AI pipeline to create incidents
- **Classifier_Lambda**: Utility Lambda function for signal classification operations
- **Incident**: A validated road infrastructure problem created by clustering and analyzing multiple related signals
- **Cluster**: A group of signals that are geographically proximate, temporally related, and semantically similar
- **Embedding**: A 1024-dimensional vector representation of signal text generated by Amazon Titan Embeddings V2
- **ChromaDB**: Vector database running on EC2 that stores signal embeddings and metadata
- **Confidence_Score**: A 0-100 numerical score indicating the likelihood that an incident represents a real road problem
- **Damage_Type**: Classification category for road issues (pothole, surface_wear, flooding, general)
- **Severity_Level**: Priority classification for incidents (low, medium, high, critical)
- **Classification_Agent**: AI agent that determines if a signal is road-related and identifies damage type
- **Intent_Agent**: AI agent that filters noise, sarcasm, and speculation from signals
- **Correlation_Agent**: AI agent that clusters signals by location, time, and semantic similarity
- **Inference_Agent**: AI agent that computes confidence scores and severity levels for clusters
- **Explanation_Agent**: AI agent that generates human-readable summaries of incidents
- **DynamoDB_Signals_Table**: DynamoDB table storing raw and processed signals with 30-day TTL
- **DynamoDB_Incidents_Table**: DynamoDB table storing validated incidents
- **S3_Bucket**: S3 bucket for signal backup and static website hosting
- **EventBridge**: AWS service that triggers Lambda functions on scheduled intervals
- **CloudFront**: CDN that serves the incident dashboard website
- **Bedrock**: AWS service providing access to foundation models (Nova Micro, Nova Lite, Titan Embeddings)

## Requirements

### Requirement 1: Automated Signal Collection

**User Story:** As a system operator, I want signals to be automatically collected from multiple data sources every hour, so that the system continuously monitors road conditions without manual intervention.

#### Acceptance Criteria

1. WHEN EventBridge triggers the hourly schedule, THE Scraper_Lambda SHALL execute within 60 seconds
2. THE Scraper_Lambda SHALL collect signals from 8 RSS news feeds (Times of India, NDTV, The Hindu, Deccan Herald, Hindustan Times, and 3 additional sources)
3. THE Scraper_Lambda SHALL collect weather data from OpenWeatherMap API for monitored cities
4. THE Scraper_Lambda SHALL load API keys from AWS Secrets Manager before scraping
5. THE Scraper_Lambda SHALL deduplicate signals by signal_id before storage
6. WHEN a signal is collected, THE Scraper_Lambda SHALL assign a unique signal_id using SHA-256 hash of content and timestamp
7. THE Scraper_Lambda SHALL collect between 5 and 10 signals per hour on average
8. WHEN scraping fails for a source, THE Scraper_Lambda SHALL log the error and continue processing other sources
9. THE Scraper_Lambda SHALL return a summary containing RSS count, weather count, total unique count, and DynamoDB write counts

### Requirement 2: Multi-Language Translation

**User Story:** As a system operator, I want non-English signals to be automatically translated to English, so that the AI pipeline can process content from all Indian languages uniformly.

#### Acceptance Criteria

1. WHEN a signal is collected, THE Scraper_Lambda SHALL detect the language using Amazon Translate
2. WHEN a signal is in Hindi, Tamil, Telugu, Kannada, Bengali, Marathi, Malayalam, Gujarati, Punjabi, or Urdu, THE Scraper_Lambda SHALL translate it to English
3. THE Scraper_Lambda SHALL store both original_content and translated_content in the signal
4. THE Scraper_Lambda SHALL store the detected_language code in the signal
5. WHEN a signal is already in English, THE Scraper_Lambda SHALL copy original_content to translated_content without calling translation API
6. WHEN translation fails, THE Scraper_Lambda SHALL use original_content as translated_content and log a warning

### Requirement 3: Signal Storage with TTL

**User Story:** As a system operator, I want signals to be stored in DynamoDB with automatic expiration, so that storage costs remain predictable and old data is automatically cleaned up.

#### Acceptance Criteria

1. WHEN a signal is ready for storage, THE Scraper_Lambda SHALL write it to DynamoDB_Signals_Table
2. THE Scraper_Lambda SHALL set a TTL attribute to 30 days from the current timestamp
3. THE Scraper_Lambda SHALL store location as a DynamoDB Map type with coordinates (lat, lon), accuracy_meters, and address fields
4. THE Scraper_Lambda SHALL convert all float values to Decimal type before writing to DynamoDB
5. THE Scraper_Lambda SHALL use conditional put with signal_id not_exists check to prevent duplicate writes
6. WHEN a duplicate signal_id is detected, THE Scraper_Lambda SHALL skip the write and increment the duplicates counter
7. THE Scraper_Lambda SHALL populate scraped_at timestamp for all signals
8. WHEN DynamoDB write fails, THE Scraper_Lambda SHALL log the error and increment the failed counter

### Requirement 4: Vector Embedding Generation

**User Story:** As a system operator, I want signal text to be converted to vector embeddings, so that semantically similar signals can be identified during correlation.

#### Acceptance Criteria

1. WHEN a new signal is inserted into DynamoDB_Signals_Table, THE DynamoDB Stream SHALL trigger Ingest_Lambda within 1 second
2. THE Ingest_Lambda SHALL only process INSERT events from the stream
3. THE Ingest_Lambda SHALL extract translated_content from the signal
4. THE Ingest_Lambda SHALL invoke Amazon Bedrock Titan Embeddings V2 model to generate a 1024-dimensional embedding vector
5. WHEN embedding generation fails, THE Ingest_Lambda SHALL log the error and return a failed status for that signal
6. THE Ingest_Lambda SHALL validate that the embedding dimension is exactly 1024 before storage
7. WHEN embedding dimension is incorrect, THE Ingest_Lambda SHALL raise a ValueError

### Requirement 5: Vector Database Storage

**User Story:** As a system operator, I want signal embeddings and metadata to be stored in ChromaDB, so that semantic similarity searches can be performed during correlation.

#### Acceptance Criteria

1. WHEN an embedding is generated, THE Ingest_Lambda SHALL store it in ChromaDB collection with signal_id as document ID
2. THE Ingest_Lambda SHALL store signal metadata including signal_type, original_content, detected_language, source, source_name, timestamp, city, latitude, longitude, accuracy_meters, and address
3. THE Ingest_Lambda SHALL remove None values from metadata before storing in ChromaDB
4. THE Ingest_Lambda SHALL use HTTP POST request to ChromaDB API endpoint at /api/v2/tenants/{tenant}/databases/{database}/collections/{collection_id}/add
5. WHEN ChromaDB connection fails, THE Ingest_Lambda SHALL raise a RuntimeError with connection details
6. WHEN ChromaDB returns HTTP error, THE Ingest_Lambda SHALL raise a RuntimeError with status code and error body
7. THE Ingest_Lambda SHALL perform a heartbeat check to ChromaDB before processing batch

### Requirement 6: Signal Enrichment and S3 Trigger

**User Story:** As a system operator, I want enriched signals to be written to S3, so that the Inference Lambda is automatically triggered to process them.

#### Acceptance Criteria

1. WHEN a signal is successfully stored in ChromaDB, THE Ingest_Lambda SHALL write the enriched signal to S3_Bucket
2. THE Ingest_Lambda SHALL add an ingested_at timestamp to the signal before writing to S3
3. THE Ingest_Lambda SHALL parse JSON string fields (location, weather_data) back to dict objects before writing to S3
4. THE Ingest_Lambda SHALL write signals to S3 with key format signals/{signal_id}.json
5. THE Ingest_Lambda SHALL set ContentType to application/json for S3 objects
6. WHEN S3 write succeeds, THE S3 ObjectCreated event SHALL trigger Inference_Lambda within 1 second
7. THE Ingest_Lambda SHALL return success status with S3 key for successfully processed signals

### Requirement 7: Signal Normalization

**User Story:** As a system operator, I want signals to be normalized before AI processing, so that all agents receive consistent data structures.

#### Acceptance Criteria

1. WHEN Inference_Lambda receives a signal from S3, THE Inference_Lambda SHALL invoke the normalizer module
2. THE normalizer SHALL ensure location field is a dict with coordinates, accuracy_meters, and address subfields
3. THE normalizer SHALL ensure coordinates contains lat and lon as float values
4. THE normalizer SHALL parse JSON string fields back to dict objects if needed
5. THE normalizer SHALL set default values for missing optional fields
6. THE normalizer SHALL validate that signal_id and translated_content are present and non-empty

### Requirement 8: Road-Related Classification

**User Story:** As a municipal authority, I want signals to be classified as road-related or not, so that only relevant infrastructure issues are processed further.

#### Acceptance Criteria

1. WHEN a normalized signal is ready, THE Classification_Agent SHALL analyze the translated_content
2. THE Classification_Agent SHALL invoke Amazon Bedrock Nova Micro model with classification prompt
3. THE Classification_Agent SHALL determine is_road_related as true or false
4. THE Classification_Agent SHALL identify damage_type as pothole, surface_wear, flooding, general, or null
5. THE Classification_Agent SHALL compute a confidence score between 0.01 and 0.99
6. THE Classification_Agent SHALL provide a reasoning string explaining the classification decision
7. WHEN a signal mentions potholes, craters, holes, gaddha, or khada, THE Classification_Agent SHALL classify damage_type as pothole
8. WHEN a signal mentions cracks, worn surface, broken asphalt, or uneven road, THE Classification_Agent SHALL classify damage_type as surface_wear
9. WHEN a signal mentions waterlogging, road underwater, or flood on road, THE Classification_Agent SHALL classify damage_type as flooding
10. WHEN a signal is road-related but does not fit specific categories, THE Classification_Agent SHALL classify damage_type as general
11. WHEN a signal source is weather, THE Classification_Agent SHALL pre-classify as road-related with damage_type flooding and confidence 0.75 without calling AI model
12. WHEN a signal location address contains "mumbai" (case-insensitive), THE Classification_Agent SHALL classify as not road-related with confidence 0.0
13. WHEN classification model response cannot be parsed, THE Classification_Agent SHALL apply fallback classification with is_road_related false and confidence 0.1
14. THE Classification_Agent SHALL achieve 92% or higher classification accuracy

### Requirement 9: Intent and Context Analysis

**User Story:** As a municipal authority, I want sarcasm, speculation, and noise to be filtered out, so that only genuine problem reports create incidents.

#### Acceptance Criteria

1. WHEN a signal is classified as road-related, THE Intent_Agent SHALL analyze the intent behind the text
2. THE Intent_Agent SHALL invoke Amazon Bedrock Nova Micro model with intent analysis prompt
3. THE Intent_Agent SHALL determine is_problem_report as true or false
4. THE Intent_Agent SHALL classify urgency_level as low, medium, high, or critical
5. THE Intent_Agent SHALL identify context_type as direct_report, indirect_mention, news_article, weather_alert, speculation, sarcasm, or ambiguous
6. THE Intent_Agent SHALL compute a confidence_modifier between -0.4 and +0.2
7. THE Intent_Agent SHALL apply the confidence_modifier to the classification confidence score
8. WHEN a signal is a first-hand account with location specifics, THE Intent_Agent SHALL apply a positive confidence_modifier between +0.1 and +0.2
9. WHEN a signal contains sarcasm or clear non-report language, THE Intent_Agent SHALL apply a negative confidence_modifier between -0.3 and -0.4
10. WHEN a signal describes immediate danger, accident, or completely blocked road, THE Intent_Agent SHALL classify urgency_level as critical
11. WHEN a signal describes significant damage or vehicle damage, THE Intent_Agent SHALL classify urgency_level as high
12. WHEN a signal source is weather, THE Intent_Agent SHALL pre-classify as problem report with urgency medium and context weather_alert without calling AI model
13. WHEN a signal source is news, THE Intent_Agent SHALL classify context_type as news_article
14. WHEN intent model response cannot be parsed, THE Intent_Agent SHALL apply fallback intent with is_problem_report true and confidence_modifier 0.0

### Requirement 10: Geospatial and Temporal Clustering

**User Story:** As a municipal authority, I want related signals to be grouped into clusters, so that multiple reports about the same issue are consolidated.

#### Acceptance Criteria

1. WHEN signals are ready for correlation, THE Correlation_Agent SHALL filter signals where is_road_related is true AND is_problem_report is true
2. THE Correlation_Agent SHALL also include weather signals regardless of is_problem_report value
3. THE Correlation_Agent SHALL fetch embeddings for all eligible signals from ChromaDB or generate them using Titan Embeddings V2
4. THE Correlation_Agent SHALL compare every pair of signals for clustering eligibility
5. WHEN two signals have timestamps within 7 days of each other, THE Correlation_Agent SHALL consider them temporally related
6. WHEN two signals have coordinates within 500 meters of each other, THE Correlation_Agent SHALL consider them geographically related
7. WHEN two signals have no coordinates but share the same city name, THE Correlation_Agent SHALL consider them geographically related
8. WHEN two signal embeddings have cosine similarity of 0.75 or higher, THE Correlation_Agent SHALL consider them semantically similar
9. WHEN two signals are temporally related AND geographically related AND semantically similar, THE Correlation_Agent SHALL group them into the same cluster
10. THE Correlation_Agent SHALL use union-find algorithm to merge overlapping signal groups
11. THE Correlation_Agent SHALL discard clusters with fewer than 1 signal
12. WHEN a cluster is formed, THE Correlation_Agent SHALL compute the geographic centroid of all signal coordinates
13. THE Correlation_Agent SHALL identify the most common damage_type in the cluster as primary_damage
14. THE Correlation_Agent SHALL extract unique source names as source_diversity list
15. THE Correlation_Agent SHALL generate a deterministic cluster_id from sorted signal IDs using SHA-256 hash

### Requirement 11: Confidence Scoring and Severity Assessment

**User Story:** As a municipal authority, I want incidents to have confidence scores and severity levels, so that I can prioritize response efforts.

#### Acceptance Criteria

1. WHEN a cluster is formed, THE Inference_Agent SHALL compute a confidence_score between 0 and 100
2. THE Inference_Agent SHALL award up to 30 points for source diversity (more diverse sources = higher score)
3. THE Inference_Agent SHALL award up to 20 points for signal count (more signals = higher score)
4. THE Inference_Agent SHALL award up to 20 points for urgency levels (higher urgency = higher score)
5. THE Inference_Agent SHALL award up to 20 points for average classification confidence
6. THE Inference_Agent SHALL award up to 10 points for signal recency (signals within 24 hours = full points)
7. THE Inference_Agent SHALL award up to 10 bonus points when cluster contains a weather signal
8. WHEN source_diversity includes 3 or more distinct sources, THE Inference_Agent SHALL add 10 bonus points
9. WHEN source_diversity includes 2 distinct sources, THE Inference_Agent SHALL add 5 bonus points
10. THE Inference_Agent SHALL clamp the final confidence_score to maximum 100
11. WHEN confidence_score is 85 or higher OR any signal has critical urgency, THE Inference_Agent SHALL set severity_level to critical
12. WHEN confidence_score is 65 or higher OR any signal has high urgency, THE Inference_Agent SHALL set severity_level to high
13. WHEN confidence_score is 45 or higher, THE Inference_Agent SHALL set severity_level to medium
14. WHEN confidence_score is below 45, THE Inference_Agent SHALL set severity_level to low
15. WHEN confidence_score is 30 or below, THE Inference_Agent SHALL not create an incident
16. THE Inference_Agent SHALL create incidents only when confidence_score exceeds 30
17. THE Inference_Agent SHALL record confidence_history with timestamp, score, and reason for each scoring event

### Requirement 12: Human-Readable Incident Explanations

**User Story:** As a municipal authority, I want incidents to have clear explanations, so that I can understand why the system flagged them without technical knowledge.

#### Acceptance Criteria

1. WHEN an incident is created, THE Explanation_Agent SHALL generate a human-readable explanation
2. THE Explanation_Agent SHALL invoke Amazon Bedrock Nova Lite model with explanation prompt
3. THE Explanation_Agent SHALL include the number and types of sources in the explanation
4. THE Explanation_Agent SHALL mention original languages when signals were translated from non-English
5. THE Explanation_Agent SHALL reference the location and damage type in the explanation
6. THE Explanation_Agent SHALL write explanations in 2-4 sentences maximum
7. THE Explanation_Agent SHALL write in plain English for non-technical municipal officials
8. THE Explanation_Agent SHALL not mention confidence scores or technical terms in explanations
9. THE Explanation_Agent SHALL not use bullet points in explanations
10. THE Explanation_Agent SHALL truncate explanations to maximum 600 characters
11. WHEN AI model fails to generate explanation, THE Explanation_Agent SHALL build a fallback explanation from structured incident data

### Requirement 13: Incident Persistence

**User Story:** As a municipal authority, I want validated incidents to be stored in a database, so that I can query and review them through the dashboard.

#### Acceptance Criteria

1. WHEN incidents are explained, THE Inference_Lambda SHALL write them to DynamoDB_Incidents_Table
2. THE Inference_Lambda SHALL store incident_id, cluster_id, damage_type, severity_level, confidence_score, status, explanation, signal_count, signal_ids, source_diversity, location, confidence_history, created_at, and updated_at fields
3. THE Inference_Lambda SHALL serialize location and confidence_history as JSON strings before writing to DynamoDB
4. THE Inference_Lambda SHALL convert Decimal values to native Python types before JSON serialization
5. THE Inference_Lambda SHALL set status to active for newly created incidents
6. THE Inference_Lambda SHALL use batch writer for efficient DynamoDB writes
7. THE Inference_Lambda SHALL remove empty string and None values before writing to DynamoDB

### Requirement 14: Scheduled Batch Processing

**User Story:** As a system operator, I want the system to periodically re-cluster all signals, so that incidents are updated with new information.

#### Acceptance Criteria

1. WHEN EventBridge triggers a scheduled event, THE Inference_Lambda SHALL detect it by checking event source field
2. WHEN running in scheduled mode, THE Inference_Lambda SHALL load all signals from DynamoDB_Signals_Table using scan operation
3. THE Inference_Lambda SHALL limit scan to 100 signals per batch
4. THE Inference_Lambda SHALL parse JSON string fields (location, classification, intent) back to dict objects
5. WHEN signals lack classification or intent fields, THE Inference_Lambda SHALL invoke Classification_Agent and Intent_Agent
6. THE Inference_Lambda SHALL update signals in DynamoDB with new classification and intent data
7. THE Inference_Lambda SHALL skip to correlation step after classification and intent are populated
8. WHEN no clusters are formed, THE Inference_Lambda SHALL return success with 0 incidents created
9. WHEN no incidents meet confidence threshold, THE Inference_Lambda SHALL return success with 0 incidents created

### Requirement 15: Error Handling and Resilience

**User Story:** As a system operator, I want the system to handle errors gracefully, so that one failure does not block processing of other signals.

#### Acceptance Criteria

1. WHEN a scraper fails for one source, THE Scraper_Lambda SHALL log the error and continue processing other sources
2. WHEN DynamoDB Stream batch contains multiple records, THE Ingest_Lambda SHALL process each record independently
3. WHEN one signal fails in Ingest_Lambda, THE Ingest_Lambda SHALL continue processing other signals in the batch
4. THE Ingest_Lambda SHALL always return HTTP 200 status to prevent Lambda retry of entire batch
5. WHEN ChromaDB is unreachable, THE Ingest_Lambda SHALL log the error and return failed status for affected signals
6. WHEN AI model invocation fails, THE agents SHALL apply fallback logic and log warnings
7. WHEN JSON parsing fails, THE agents SHALL use default values and log warnings
8. THE Inference_Lambda SHALL wrap S3 read operations in try-except blocks
9. THE Inference_Lambda SHALL wrap DynamoDB write operations in try-except blocks
10. WHEN Inference_Lambda encounters an error, THE Inference_Lambda SHALL return HTTP 500 with error details

### Requirement 16: Performance and Cost Optimization

**User Story:** As a system operator, I want the system to operate efficiently, so that monthly costs remain under $35.

#### Acceptance Criteria

1. THE system SHALL process 5-10 signals per hour on average
2. THE system SHALL create 200-400 incidents per month on average
3. THE system SHALL maintain 99.9% uptime
4. THE system SHALL complete Scraper_Lambda execution within 30 seconds on average
5. THE system SHALL complete Ingest_Lambda execution within 5 seconds per signal
6. THE system SHALL complete Inference_Lambda execution within 60 seconds for single signal processing
7. THE Classification_Agent SHALL pre-classify weather signals without calling AI models to save costs
8. THE Classification_Agent SHALL filter Mumbai signals without calling AI models to save costs
9. THE Intent_Agent SHALL pre-classify weather and news signals without calling AI models when possible
10. THE system SHALL incur total monthly operational costs between $25 and $35

### Requirement 17: Data Quality and Accuracy

**User Story:** As a municipal authority, I want the system to maintain high accuracy, so that I can trust the incident reports.

#### Acceptance Criteria

1. THE system SHALL achieve 92% or higher classification accuracy
2. THE system SHALL maintain false positive rate below 8%
3. THE system SHALL deduplicate signals by signal_id to prevent duplicate processing
4. THE system SHALL validate signal structure before processing
5. WHEN translated_content is empty, THE agents SHALL use original_content as fallback
6. WHEN coordinates are missing, THE agents SHALL use city-level matching for clustering
7. WHEN timestamp is missing or invalid, THE agents SHALL assume signals are within time window
8. THE system SHALL preserve original_content and detected_language for audit purposes

### Requirement 18: Monitoring and Observability

**User Story:** As a system operator, I want detailed logs and metrics, so that I can monitor system health and debug issues.

#### Acceptance Criteria

1. THE Scraper_Lambda SHALL log RSS count, weather count, unique count, written count, duplicates count, and failed count
2. THE Ingest_Lambda SHALL log processed count, succeeded count, failed count, and individual signal results
3. THE Inference_Lambda SHALL log signals processed, clusters formed, and incidents created
4. THE Classification_Agent SHALL log signal_id, is_road_related, damage_type, and confidence for each classification
5. THE Intent_Agent SHALL log signal_id, is_problem_report, urgency_level, context_type, and confidence adjustment
6. THE Correlation_Agent SHALL log eligible signal count, cluster count, signal count per cluster, and source diversity
7. THE Inference_Agent SHALL log incident_id, confidence_score, severity_level, and damage_type for each incident
8. THE Explanation_Agent SHALL log incident_id and explanation character count
9. THE agents SHALL log warnings for fallback logic activation
10. THE agents SHALL log errors with exception details using exc_info=True

### Requirement 19: AWS Infrastructure Configuration

**User Story:** As a system operator, I want the system to use AWS managed services, so that infrastructure maintenance is minimized.

#### Acceptance Criteria

1. THE Scraper_Lambda SHALL be triggered by EventBridge on an hourly schedule
2. THE Ingest_Lambda SHALL be triggered by DynamoDB Streams on DynamoDB_Signals_Table
3. THE Inference_Lambda SHALL be triggered by S3 ObjectCreated events on S3_Bucket
4. THE Inference_Lambda SHALL also be triggered by EventBridge on a scheduled interval for batch processing
5. THE DynamoDB_Signals_Table SHALL have signal_id as partition key
6. THE DynamoDB_Signals_Table SHALL have DynamoDB Streams enabled with NEW_IMAGE view type
7. THE DynamoDB_Signals_Table SHALL have TTL enabled on ttl attribute
8. THE DynamoDB_Incidents_Table SHALL have incident_id as partition key
9. THE S3_Bucket SHALL have event notifications configured to trigger Inference_Lambda on ObjectCreated events
10. THE S3_Bucket SHALL be configured as origin for CloudFront distribution
11. THE CloudFront distribution SHALL serve the static incident dashboard website
12. THE Lambda functions SHALL have IAM roles with permissions for DynamoDB, S3, Bedrock, Secrets Manager, and Translate
13. THE Lambda functions SHALL use Python 3.12 or higher runtime
14. THE ChromaDB instance SHALL run on EC2 with HTTP API exposed on port 8000

### Requirement 20: Security and Access Control

**User Story:** As a system operator, I want API keys and credentials to be securely managed, so that sensitive data is not exposed.

#### Acceptance Criteria

1. THE Scraper_Lambda SHALL load API keys from AWS Secrets Manager before scraping
2. THE Scraper_Lambda SHALL check environment variables first before calling Secrets Manager
3. THE Secrets Manager secret SHALL contain OPENWEATHER_API_KEY and other required API keys
4. THE Lambda functions SHALL use IAM roles for AWS service authentication
5. THE Lambda functions SHALL not store credentials in code or environment variables
6. THE DynamoDB tables SHALL use encryption at rest
7. THE S3_Bucket SHALL use encryption at rest
8. THE ChromaDB instance SHALL be accessible only from Lambda function security group
